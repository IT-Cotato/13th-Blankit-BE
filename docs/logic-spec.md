## 1. 우선순위 추천 로직

과업의 우선순위(상/중/하)를 계산하는 로직. `task.priority` 값이 이 로직으로 결정된다.

### 1-1. 기본 순위: 긴급도 + 중요도 2차 정렬

1. **1차 정렬: 긴급도** — 마감일까지 남은 일수가 적은 순
2. **2차 정렬: 중요도** — 남은 일수가 같으면 중요 표시(`is_starred`)한 과업 우선

예시:

| 과업 | 남은 일수 | 중요 여부 | 기본 순위 |
| --- | --- | --- | --- |
| 과제 제출 | 2일 | X | 1 |
| 기말고사 | 7일 | O | 2 |
| 영어 단어 | 7일 | X | 3 |
| 자격증 | 30일 | O | 4 |

### 1-2. 진행 부족 순위

모든 과업을 진행률이 낮은 순으로 정렬하여 순위를 매긴다.

| 과업 | 진행률 | 진행 부족 순위 |
| --- | --- | --- |
| 영어 단어 | 0% | 1 |
| 자격증 | 10% | 2 |
| 기말고사 | 20% | 3 |
| 과제 제출 | 60% | 4 |

### 1-3. 점수 계산

```
점수 = 기본 순위 × 0.8 + 진행 부족 순위 × 0.2
```

점수가 **낮을수록** 우선순위가 높다. (`daily_recommendation_item.score`에 저장)

### 1-4. 상/중/하 배분 — 과업 3개 이상

점수가 낮은 순으로 정렬한 뒤:

1. 상위 30% → 우선순위 **상**
2. 다음 40% → 우선순위 **중**
3. 나머지 → 우선순위 **하**
4. 30%/40% 계산 시 소수점은 **반올림**

```
high_end = round(total_tasks * 0.3)    # '상' 경계
medium_end = round(total_tasks * 0.7)  # '중' 경계

if rank <= high_end:
    priority = "상"
elif rank <= medium_end:
    priority = "중"
else:
    priority = "하"
```

### 1-5. 상/중/하 배분 — 과업 2개 이하

| 과업 개수 N | 상 | 중 | 하 |
| --- | --- | --- | --- |
| 1 | 1 | 0 | 0 |
| 2 | 1 | 1 | 0 |

## 2. 전체 진행률 계산 방식

사용자가 하나의 과업을 여러 세부 단계(`task_step`)로 분할한 경우, 전체 진행률은 세부 단계 개수 기준으로 **균등 가중** 계산한다. 이때 전체 진행률은 사용자가 직접 조정할 수 없다(자동 계산, functional-spec 2.9.4).

### 계산식

```
전체 진행률 = Σ ( 각 세부단계 진행률 × (1 / 세부단계 개수) )
```

### 예시

**세부 단계 3개인 경우** (단계당 비중 = 100% ÷ 3 ≒ 33%)

- 1개 완료(100%) → 전체 33%
- 2개 완료 → 전체 67%
- 3개 완료 → 전체 100%

**세부 단계 5개인 경우** (단계당 비중 = 20%)

- 1개 완료 → 20%, 2개 → 40%, 3개 → 60%, 4개 → 80%, 5개 → 100%

## 3. 예상 시간 보정 계산 방법

`task.estimated_time`(남은 예상 시간)을 피드백 시점마다 갱신하는 로직.
"비슷한 과업"은 과업 등록 시 사용자가 선택한 `task.similar_task_id`가 가리키는 과거 과업이다.

### 3-1. 초기 사용자 (비슷한 과거 과업 데이터가 없을 때)

```
남은 예상시간 = (사용시간 / 진행률) × 남은 진행률
```

예시: 2시간 동안 40%를 진행한 경우 → (2시간 / 40%) × 60% = **3시간**

### 3-2. 일반 사용자 (비슷한 과거 과업 데이터가 있을 때)

시나리오 예시: 피드백으로 진행률을 10% → 50% → 75% 순서로 갱신하는 경우.

#### 핵심 공식

```
새 남은 예상시간 = A + B + (C × 보정계수 × D)
```

#### 첫 피드백 시점 (진행률 10% 입력)

- 과거 비슷한 과업의 0-10% 구간 시간을 가져옴 (예: 3시간)
- 현 과업의 0-10% 실제 소요 시간을 가져옴 (예: 5시간)

| 항 | 정의 |
| --- | --- |
| A (기존 예상 시간) | 비슷한 과업의 총 소요 시간 − 현재 소요한 시간 |
| B (누적 시간 차이) | 현재 0-10% 소요 시간 − 과거 0-10% 소요 시간 |
| C (누적 속도 차이) | 현 과업의 0-10% 속도 − 과거 과업의 0-10% 속도 |
| D (남은 진행률) | 90% |

#### 두 번째 피드백 시점 (진행률 50% 입력)

| 항 | 정의 |
| --- | --- |
| A (기존 남은 예상 시간) | 직전 피드백(10%) 시점에 계산된 남은 예상 시간 − (10-50%) 구간 소요 시간 |
| B ((10-50%) 구간 소요 시간 차이) | (현재 0-50% 소요 − 과거 0-50% 소요) − (현재 0-10% 소요 − 과거 0-10% 소요) |
| C ((10-50%) 구간 속도 차이) | [(0-50%) 현 과업 속도 − (0-50%) 과거 과업 속도] − [(0-10%) 현 과업 속도 − (0-10%) 과거 과업 속도] |
| D (남은 진행률) | 50% |

이후 피드백도 같은 방식으로 "직전 피드백 구간"을 기준으로 반복한다.

### 3-3. 보정계수 조절

- **지속적으로** 지연되거나 지속적으로 단축되는 경우 → 보정계수를 **늘림**
- **일시적** 지연 또는 단축 → 보정계수를 **줄임**

방법: 연속성 카운트 변수 `count` 사용. 구간 시간 차이(B)의 부호(느림/빠름)가 직전과 같으면 count+1, 부호가 바뀌면 count를 1로 리셋.

예시 — 계속 느린 경우:

| 구간 B | diff | count |
| --- | --- | --- |
| +5 (0-10% 차이) | 느림 | 1 |
| +3 (10-50% 차이) | 느림 | 2 |
| +7 (50-75% 차이) | 느림 | 3 |
| -3 (75-90% 차이) | 빠름 | 1 (리셋) |

예시 — 계속 빠른 경우:

| 구간 B | diff | count |
| --- | --- | --- |
| -5 (0-10% 차이) | 빠름 | 1 |
| -3 (10-50% 차이) | 빠름 | 2 |
| -7 (50-75% 차이) | 빠름 | 3 |
| +3 (75-90% 차이) | 느림 | 1 (리셋) |

예시 — 지연/가속 반복: +5(느림,1) → -3(빠름,1) → +7(느림,1) → -3(빠름,1)

보정계수 계산식:

```
보정계수 = count / (count + 2)
```

(유리 함수 형태라 count가 커져도 1을 넘지 않고 수렴)

### 3-4. 과거 과업의 구간 시간을 가져오는 방법 (피드백 시점 불일치 해결)

**문제**: 현 과업은 10%, 30%, 50%마다 피드백했는데 과거 과업은 20%, 45%, 60%에 피드백한 경우, "0-10%에 해당하는 과거 시간"이 직접 존재하지 않음.

**해결**: 과거 과업의 피드백 구간별로 `1%당 평균 소요 시간`을 계산해두고, 현 과업의 진행률이 속한 과거 구간의 속도를 적용해 환산한다.

과거 과업 구간별 속도 예시:

| 과거 구간 | 구간 시간 | 1%당 속도 |
| --- | --- | --- |
| 0-20% | 2시간 | 6분 |
| 20-45% | 4시간 | 4분 6초 |
| 45-60% | 10시간 | 2분 6초 |

환산 예시 (현 과업이 10%, 30%, 50%마다 피드백한 경우):

| 현 과업 기준 | 과거 시간 환산 | 근거 |
| --- | --- | --- |
| 0-10% | 6분 × 10 | 현 과업의 10%는 과거의 0-20% 구간에 해당 → 그 구간 속도 × 10 |
| 0-30% | 4분 6초 × 30 | 현 과업의 30%는 과거의 20-45% 구간에 해당 → 그 구간 속도 × 30 |
| 0-50% | 2분 6초 × 50 | 현 과업의 50%는 과거의 45-60% 구간에 해당 → 그 구간 속도 × 50 |

### 3-5. 예외 처리

새 남은 예상시간이 **음수**가 되는 경우 → **30분으로 교체**

## 4. 과업 조합 추천 로직

과업 조합 추천 모드별 로직. 색상은 우선순위를 의미한다: 빨간색=상, 노란색=중, 초록색=하.
(모드 코드값 매핑: 불끄기=FIRE, 밸런스=BALANCE, 찍먹=TASTE, 해치우기=CLEAR — `playlist_item.source_mode`, `daily_recommendation.mode`에 사용)

### 4-1. 불끄기 모드 (FIRE)

오늘 최소 시간을 오직 **빨간색(상) 과업**에만 올인하는 조합.

1. 빨간색 과업 중 우선순위 1등의 남은 예상 시간이 오늘의 최소 시간 **이상**이면 → 그 과업 1개만 추천
2. 1등의 남은 예상 시간이 오늘의 최소 시간 **미만**이면 → 남은 시간을 2순위 빨간색 과업에 배치

예시 (오늘 최소 시간 3시간):
- 1등 빨간 과업이 3시간 이상 남음 → 그 과업 1개에 3시간 통으로 할당
- 1등이 2시간이면 → 남은 1시간을 2등 빨간 과업에 할당

### 4-2. 밸런스 모드 (BALANCE)

**노란색/초록색 과업 1개 + 빨간색 과업 1개** 조합.

- 노란색/초록색 중 남은 예상 시간이 가장 적은(가장 빨리 끝나는) 과업 + 우선순위가 가장 높은 빨간색 과업
- 의도: 성취감을 먼저 제공해 부담 높은 빨간색 과업의 진입 장벽을 낮춤

### 4-3. 찍먹 모드 (TASTE)

각 색상 영역의 1등 과업을 하나씩 조합.

- 빨간색 1등 + 노란색 1등 + 초록색 1등

### 4-4. 해치우기 모드 (CLEAR)

색상과 무관하게 빨리 끝낼 수 있는 과업 위주 배치.

1. 모든 과업을 '마감까지 남은 시간'이 짧은 순(오름차순)으로 정렬
2. 1순위 과업의 예상 시간이 오늘 가용 시간보다 **긴** 경우 → 조합 없이 그 과업 1개만
3. 1순위 과업의 예상 시간이 오늘 가용 시간보다 **짧은** 경우 → 1순위를 먼저 배치한 뒤, 남은 시간 안에 들어오는 다음으로 마감이 급한 과업을 조합

## 5. 오늘 최소 시간 로직 (= 오늘의 권장 시간)

홈 화면의 "오늘의 권장 시간"(functional-spec 2.2) 계산 로직.
`daily_recommendation.total_recommended_minutes`에 저장.

### 계산식

```
오늘의 권장 시간 = Σ ( 과업별 남은 예상 시간 ÷ 마감일까지 남은 일수 )
```

예시: '알고리즘 시험'의 남은 예상 시간 5시간, 남은 일수 5일 → 5시간 ÷ 5일 = 하루 1시간.
이런 과업별 하루 권장 시간을 전부 합산한 값이 오늘의 권장 시간.
